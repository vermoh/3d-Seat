01Контекст

iTicket использует WebGL для 3D-визуализации залов. Пользователь видит зал в перспективе, вращает камеру, увеличивает сектора, кликом выбирает место. Параллельно работает 2D-режим — быстрее и точнее для опытных пользователей.

Аналогичные решения на рынке: 3D Digital Venue, Seats3D (Ballena), Virtual Venue от Ticketmaster. Принцип одинаковый: 3D-художник строит модель зала, движок рендерит её в браузере, бэкенд отдаёт инвентарь мест.

Задача — встроить подобный модуль в существующий билетный сервис. Это не «нарисовать 3D», а спроектировать систему, которая:

  • получает структуру зала и состояние мест из бэкенда;
  • рендерит её в двух режимах с переключением;
  • обрабатывает выбор, временную блокировку и продажу;
  • деградирует на слабых устройствах;
  • не ломает существующий чекаут — заменяет один шаг.
Главный тезис

70% стоимости системы — не в 3D-движке, а в данных и интеграции. Сам рендер пишется за несколько недель. Основной объём работ — создание моделей залов, инвентарь, блокировки, синхронизация с CRM, поддержка нескольких площадок.

03Модель зала

Фундамент системы. Ошибка в модели заставит переделывать весь стек. Базовая иерархия:

// venue.json
{
  "id": "hall-001",
  "name": "Концертный зал",
  "stage": { "x": 0, "y": 0, "z": -2, "width": 14 },
  "sections": [
    {
      "id": "parter",
      "label": "Партер",
      "origin": { "x": 0, "y": 0, "z": 2 },
      "rotation": 0,
      "rows": [
        {
          "index": 1,
          "seats": [
            { "id": "P-1-1", "num": 1, "category": "vip", "x": -6.6, "z": 0 }
          ]
        }
      ]
    }
  ],
  "categories": [
    { "id": "vip", "price": 250, "color": "#c9a961" }
  ]
}

Ключевые решения

Координаты: реальные или абстрактные. Реальные метры дают точные расстояния и работают в 3D. Абстрактные сетки проще, но плохо переносятся в перспективу. Если нужен 360°-вид с конкретного места — только метры.

Места: объекты или маска. Для залов до 3000 мест каждое место — отдельный объект. Для стадионов 50 000+ мест сектора хранят как растровые маски с lookup-таблицей.

Версионирование залов. Зал меняется: убрали ряд, добавили VIP-ложу. Билеты, проданные до изменения, должны указывать на старую версию схемы. Решение — поле venue_version и неизменяемые snapshot'ы.

Типовая ошибка

Смешивают модель зала (геометрия) и инвентарь (что в продаже). Это разные сущности с разным жизненным циклом. Зал стабилен, инвентарь меняется с каждым событием. Бэкенд отдаёт их отдельно, клиент склеивает.

04Рендер: 2D, 3D, 360°

В индустрии устоялись три уровня визуализации. Зрелый продукт поддерживает все три.

УровеньТехнологияНазначение
2D-схемаSVG или canvasОсновной инструмент выбора. Быстро, точно, работает на mobile. 95% покупок происходит здесь.
3D-перспективаWebGL (Three.js, Babylon.js)Помогает оценить расположение секторов относительно сцены. Даёт прирост конверсии за счёт уверенности.
360°-вид с местаPannellum, MarzipanoПанорама с конкретной точки зала. Снимается фотографом или рендерится из 3D-сцены.

Технические требования к 3D

Instanced rendering. Зал на 3000 мест — это 3000 одинаковых кресел. Отдельный draw call на каждое положит мобилку. Решение — THREE.InstancedMesh: одна геометрия, один материал, тысячи трансформаций в одном вызове GPU.

LOD (Level of Detail). На дальнем зуме кресло — куб. На ближнем — кресло с подлокотниками. Минимум два уровня, переключаются по расстоянию до камеры.

Raycasting для кликов. Клик в 3D — луч из камеры в сцену. Three.js даёт Raycaster. С instanced mesh используется intersectObject, из пересечения берётся instanceId.

Touch controls. OrbitControls работает на mobile плохо. Пишут свои жесты: один палец — вращение, два — pinch zoom, тап без движения — выбор. Порог «без движения» — 4–6 пикселей.

Performance budget. Бюджетный Android — 30 FPS. iPhone — 60. При просадке FPS — fallback на 2D с уведомлением.

Деградация

3D — не единственный режим. На старых устройствах WebGL не запустится или будет работать медленно. 2D — базовый режим, 3D — улучшение. Не наоборот.

05Загрузка моделей зала

Геометрия зала — статичная 3D-сцена: стены, потолок, сцена, декор, балконы. Её рендерят один раз при открытии страницы. Места не являются частью этой модели — они описываются в venue.json и инстансятся программно через InstancedMesh. Если рисовать каждое место отдельной 3D-моделью, производительность падает кратно.

На практике загружают модель окружения, а места строят из координат поверх.

Форматы 3D-моделей для веба

ФорматСодержаниеНазначение
GLTF / GLBПромышленный стандарт. Геометрия, текстуры, PBR-материалы, анимации в одном файле. GLB — бинарный, GLTF — JSON + ассеты.Дефолтный выбор для продакшена. Малый размер, быстрая загрузка.
OBJ + MTLПростой текстовый формат с материалами. Без анимаций и PBR.Прототипы, импорт из старого софта. В продакшен не идёт.

В 95% случаев — GLB. Один бинарный файл, один HTTP-запрос, сжатие через Draco или Meshopt, поддержка PBR.

Источники моделей

1 · Художник в Blender / 3ds Max / Maya

Основной путь. Художник получает план зала (PDF, чертёж, фото) → строит модель → экспортирует в GLB → инженер подгружает. Цикл — 3–7 дней на зал среднего размера.

2 · Готовые модели

Sketchfab, Poly Haven, CGTrader. Подходит для прототипа или для unbranded залов. Обязательная проверка лицензии: CC0, royalty-free или commercial.

3 · Photogrammetry или LIDAR-сканирование

Реальный зал снимается, на выходе — 3D-меш. Точно, но файл тяжёлый, требует чистки. Оправдано для уникальных площадок: исторические театры, концертные залы с архитектурной ценностью.

4 · Программная генерация

Зал не моделируется художником, а строится из параметров: количество рядов, кривизна, наклон, размер сцены. Подходит для типовых залов и сетей с большим числом одинаковых площадок (кинотеатры, корпоративные ивенты). Так работает прототип в текущем артефакте.

Подключение GLTFLoader

Three.js не загружает 3D-файлы из коробки. Нужен отдельный лоадер из three/examples/jsm/loaders.

// 1. Импорт
import { GLTFLoader } from 'three/examples/jsm/loaders/GLTFLoader.js';

// 2. В useEffect после создания scene:
const loader = new GLTFLoader();
loader.load(
  '/assets/venues/hall-001.glb',
  (gltf) => {
    // gltf.scene — THREE.Group со всей геометрией зала
    gltf.scene.position.set(0, 0, 0);
    gltf.scene.scale.set(1, 1, 1);
    scene.add(gltf.scene);
  },
  (xhr) => console.log(`${(xhr.loaded / xhr.total) * 100}% loaded`),
  (error) => console.error('GLTF load error:', error)
);

После загрузки модель встаёт фоном, места из venue.json рисуются поверх через InstancedMesh. Координаты в venue.json (в метрах) должны соответствовать масштабу модели — это согласовывается на этапе экспорта из Blender.

Draco-компрессия

GLB можно сжать через Draco — сторонний кодек геометрии. Размер модели падает в 5–10 раз. Декодирование идёт через WebAssembly на клиенте.

import { DRACOLoader } from 'three/examples/jsm/loaders/DRACOLoader.js';

const dracoLoader = new DRACOLoader();
dracoLoader.setDecoderPath('https://www.gstatic.com/draco/v1/decoders/');
loader.setDRACOLoader(dracoLoader);

// дальше тот же loader.load()

Для модели зала на 5–15 МБ полигонов — обязательная мера. WASM-декодер берётся с Google CDN или хостится самостоятельно.

Схема данных с моделью

В venue.json добавляется поле modelUrl с путём к GLB. Клиент сначала параллельно загружает venue и модель, затем рендерит сцену.

{
  "id": "hall-001",
  "name": "Концертный зал",
  "modelUrl": "/assets/venues/hall-001.glb",
  "modelScale": 1.0,
  "modelOffset": { "x": 0, "y": 0, "z": 0 },
  "stage": { ... },
  "sections": [ ... ]
}

LOD: несколько уровней детализации

На продакшене готовят 2–3 версии модели:

  • low — упрощённая геометрия, без декора. 1–2 МБ. Для мобильных и слабых устройств.
  • medium — основная модель. 3–8 МБ. Дефолт для большинства десктопов.
  • high — детализированная, с текстурами высокого разрешения. 10–20 МБ. Опция для мощных машин.

Уровень выбирается по результатам benchmark на клиенте: измеряется FPS на тестовой сцене, дальше грузится подходящий LOD. Three.js даёт встроенный класс THREE.LOD для динамического переключения по расстоянию до камеры.

Asset pipeline в продакшене

На системе с десятками залов хранение моделей превращается в отдельную задачу.

┌─────────────────────────────────────────────────────────┐ │ Художник (Blender) → экспорт GLB → загрузка в CMS │ └──────────────────────┬──────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Asset processing pipeline │ │ • Draco-компрессия │ │ • Генерация LOD (low/medium/high) │ │ • Валидация: размер, полигоны, текстуры │ └──────────────────────┬──────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ S3 / CDN │ │ /venues/hall-001/v3/low.glb │ │ /venues/hall-001/v3/medium.glb │ │ /venues/hall-001/v3/high.glb │ └──────────────────────┬──────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Клиент: GLTFLoader → DRACOLoader → render │ └─────────────────────────────────────────────────────────┘
Версионирование

Модель зала меняется со временем. Билеты, проданные до изменения, должны рендериться с той версией, что была актуальна на момент продажи. Хранить как неизменяемые snapshot'ы: /venues/hall-001/v3/..., /venues/hall-001/v4/.... Поле venue_version в venue.json ссылается на конкретную версию.

Минимум для прототипа

В текущий артефакт seat-selector.jsx модель добавляется в 4 шага:

  1. Импортировать GLTFLoader.
  2. В useEffect компонента SeatMap3D вызвать loader.load(modelUrl, ...).
  3. В коллбэке добавить gltf.scene в scene.
  4. Удалить программные примитивы (floor, stage) — они станут частью загруженной модели.

Fallback на программную геометрию имеет смысл оставить: при ошибке загрузки или отсутствии modelUrl в данных зала прототип работает как сейчас.

Ограничение среды артефактов

В песочнице, где запущен прилагаемый прототип, динамическая загрузка subpath-модулей Three.js (three/examples/jsm/loaders/...) заблокирована. Поэтому в артефакте используется только программная геометрия зала. В обычной React-среде (Vite, Next.js, CRA) загрузка GLB через GLTFLoader работает стандартно, без обходных решений.

06Realtime: блокировка мест

Слабое место большинства самописных решений. Сценарий двойной продажи:

  1. Пользователь A открывает зал. Место 5-12 свободно.
  2. Пользователь B открывает тот же зал одновременно.
  3. A кликает 5-12. Место уходит в hold на 10 минут.
  4. B должен мгновенно увидеть, что место занято.
  5. A не оплатил за 10 минут. Место возвращается в продажу, B видит обновление.

Без realtime получается двойная продажа.

Hold-механизм

// Состояния
available  // свободно
held       // в корзине у кого-то, TTL ~10 мин
reserved   // оплата в процессе
sold       // продано

// API
POST   /events/{id}/holds        { seat_ids: [...] }  → hold_token + expires_at
DELETE /holds/{hold_token}                            → освободить
POST   /holds/{hold_token}/confirm                    → перевести в reserved

Транспорт обновлений

ПодходПлюсыМинусы
Polling N секПросто, работает вездеЛаг 2–5 сек, лишняя нагрузка
Server-Sent EventsОднонаправленный поток, проще WSТолько server → client
WebSocketДвунаправленный, мгновенныйСложнее инфраструктура, держать соединения

На MVP — polling 3–5 секунд для открытого зала. На зрелую версию — WebSocket с подпиской на event.

Тонкость

Не слать каждое изменение всем подключённым. Слать дельту по событию: «эти места изменили статус». На клиенте — patch текущего состояния, не полная перезагрузка. Иначе на ажиотажном событии бэкенд ляжет.

07Точки интеграции

Глубина интеграции определяется тремя вариантами.

A. Drop-in widget

Модуль — чёрный ящик. Бэкенд хоста отдаёт venue.json и inventory.json по согласованному формату. Виджет показывает зал и возвращает seat_ids. Корзину, оплату, билеты делает существующий сервис.

Путь по умолчанию. Минимум переделок, максимум переиспользования.

// Минимальный API между виджетом и хостом
<SeatSelector
  venueUrl="/api/venues/123/v2"
  inventoryUrl="/api/events/456/inventory"
  holdsApi="/api/events/456/holds"
  onSelectionChange={(seats) => {...}}
  onCheckout={(holdToken) => router.push('/checkout')}
/>

B. Tight integration

Виджет использует внутренние сервисы хоста: его аутентификацию, очередь, CRM. Подходит, если бэкенд хоста развит и не нужно дублировать holds на стороне виджета.

Сложнее. Даёт цельный UX — пользователь не «выпадает» в отдельный модуль.

C. Полная замена шага выбора

Виджет заменяет экран выбора целиком — со своей корзиной и своими holds. Существующий сервис подключается только на чекауте и приёме платежа.

Имеет смысл, если существующий выбор устарел и переписать всё проще, чем интегрировать.

Требования к хосту (минимум)

  • Эндпоинт инвентаря: статусы мест для события.
  • Эндпоинт holds: создание, отмена, подтверждение блокировки.
  • Цены и категории: матрица event × category → price. С учётом промокодов и динамического ценообразования.
  • Авторизация: гость или залогиненный, лимиты на покупку.

Альтернатива

Если задача коммерческая, сравните стоимость с лицензией 3D Digital Venue или аналогов. Готовые решения интегрируются за 4–6 недель. Свой стек выгоден при большом потоке событий, необходимости брендированного UX или стратегической цели владеть технологией.

09Риски

1 · Создание моделей залов

Самая недооценённая часть. Один зал на 1500 мест — 3–7 дней работы 3D-художника. 50 площадок — полгода чистой работы. Без хорошего venue editor процесс не масштабируется.

2 · Расхождение модели и реальности

Художник нарисовал зал «как должно быть». На площадке убрали два ряда, перенесли пульт, переставили колонны. Билет продан на место, которого нет. Нужен процесс верификации перед публикацией.

3 · Производительность на старых устройствах

WebGL на устройствах старше 5 лет работает плохо. На массовой аудитории нужна автоматическая деградация: низкий FPS — переключение на 2D, 3D в сессии больше не предлагается.

4 · SEO

3D-canvas не индексируется. Если страница должна попадать в выдачу, нужен SSR-fallback с 2D-схемой и meta-разметкой.

5 · Лицензии на 3D-ассеты

Текстуры дерева, кресел, освещения требуют проверки лицензии. Бесплатные ассеты — только CC0 или royalty-free. Иначе через год придёт письмо.

6 · Доступность (a11y)

3D-карта недоступна для screen reader. В ЕС и UK есть законодательство, требующее доступности. Решение: 2D-режим полностью клавиатурно-навигабельный с aria-labels на местах.

Минимальный стек для MVP

React, Three.js, SVG, REST polling, Redis для holds. WebSocket, GLTF-ассеты, venue editor и 360° — второй виток.

Прототип

В файле seat-selector.jsx реализованы слои 2 (Selection state) и 3 (Render engine): 2D SVG, 3D WebGL с переключением, выбор, корзина, ценовые категории, hover-состояния, instanced rendering, custom touch controls. Это не production-код, а демонстрация архитектурных решений. Бэкенд, realtime, holds и venue editor — за пределами прототипа.
